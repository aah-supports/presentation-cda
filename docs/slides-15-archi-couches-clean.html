Architecture en couches vs Clean Architecture

Cours TypeScript appliqué au cas d'un panier.

Objectif du chapitre

  • Comprendre la différence entre architecture en couches, hexagonale et clean.
  • Savoir choisir une architecture selon le contexte projet.
  • Relier la théorie à des exemples TypeScript concrets.

Tableau comparatif

Architecture Principe Exemple TypeScript Atout principal Limite principale
En couches Organisation par couches horizontales. Controller → Service → Repository Simple, rapide pour démarrer un CRUD. Couplage possible au framework/à la DB.
Hexagonale Métier au centre + ports/adaptateurs. UseCase + Port + Adapter Technos remplaçables facilement. Plus de structure initiale.
Clean Dépendances orientées vers le cœur métier. domain/application/interfaces/infrastructure Très bonne testabilité et évolutivité. Peut sembler verbeuse sur petit projet.

Différence clé

Architecture en couches

Organise surtout la technique: web, service, données.

Hexagonale / Clean

Organise pour protéger le métier des détails techniques.

Règle: les dépendances pointent vers domain.

Exemple TypeScript en couches

class CartController {
  constructor(private readonly service: CartService) {}

  async addItem(input: { userId: string; productId: string; qty: number }) {
    return this.service.addItem(input.userId, input.productId, input.qty);
  }
}

class CartService {
  constructor(private readonly repo: CartRepository) {}

  async addItem(userId: string, productId: string, qty: number) {
    if (qty <= 0) throw new Error("QUANTITY_INVALID");
    const cart = await this.repo.findByUserId(userId);
    cart.items.push({ productId, qty });
    await this.repo.save(cart);
    return cart;
  }
}

Exemple TypeScript clean/hexagonal

class Cart {
  constructor(public readonly userId: string, public readonly items: Array<{ productId: string; qty: number }> = []) {}

  addItem(productId: string, qty: number) {
    if (qty <= 0) throw new Error("QUANTITY_INVALID");
    this.items.push({ productId, qty });
  }
}

interface CartRepository {
  findByUserId(userId: string): Promise;
  save(cart: Cart): Promise;
}

class AddItemToCart {
  constructor(private readonly repo: CartRepository) {}
  async execute(input: { userId: string; productId: string; qty: number }) {
    const cart = await this.repo.findByUserId(input.userId);
    cart.addItem(input.productId, input.qty);
    await this.repo.save(cart);
    return cart;
  }
}

MVC, c'est quoi dans tout ça ?

  • MVC est un pattern de présentation, pas une architecture complète d'entreprise à lui seul.
  • Dans la pratique, on l'intègre souvent dans une architecture en couches.
  • Exemple courant: Vue/Controller (MVC) + Service + Repository.

Frameworks qui utilisent souvent les couches

Angular

Composants (présentation), services (logique), API services (accès données).

React (organisation projet)

Composants, hooks/services métier, clients API.

Spring Boot

Controller → Service → Repository très répandu.

ASP.NET / Laravel

Souvent enrichis d'une couche service/repository autour du MVC.

Types d'applications: en couches vs clean

Architecture en couches

Back-office, ERP/CRM interne, API CRUD, e-commerce simple.

Clean Architecture

Domaines métier complexes: e-commerce avancé, finance, santé, plateformes multi-canaux, applications à forte longévité.

Choisir simplement

  • Projet court et très CRUD: couches.
  • Métier riche + besoin de tests rapides + évolutivité: clean/hexagonale.
  • Le plus important: le sens des dépendances, pas seulement les noms de dossiers.
