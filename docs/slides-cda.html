Introduction CDA

  • Le titre a évolué de 2018 vers 2023 avec une logique sécurité renforcée.
  • Bloc 1: développer une application sécurisée.
  • Bloc 2: concevoir/développer une application en couches.
  • Bloc 3: préparer le déploiement sécurisé.

Évolutions métier mises en avant

  • Montée en puissance des pratiques Agile, CI/CD, conteneurisation.
  • Prise en compte de la résilience informatique et de l'accessibilité.
  • Compétences de communication en français/anglais attendues.

Abstract: rôle central du support

Le fichier Abstract structure la certification: livrables, méthodologie attendue, épreuve orale.

  • Le jury attend une démarche complète, pas une simple démo.
  • Sécurité et qualité doivent traverser tout le projet.
  • Le dossier projet est le document principal de certification.

3 activités du titre CDA

  • Concevoir/développer des composants d'interface sécurisés.
  • Concevoir/développer la persistance des données sécurisée.
  • Concevoir/développer une application multicouche répartie sécurisée.

Livrables attendus

Dossier professionnel (DP)

Document ministériel alimenté régulièrement avec des expériences réelles.

Dossier projet

Document principal de certification (volumétrie conséquente, démarche complète).

Dossier projet: plan type (1/2)

  • Compétences du référentiel couvertes.
  • Résumé du projet en anglais (200-250 mots).
  • Cahier des charges / expression des besoins.
  • Gestion de projet (planning, suivi, environnement humain/technique, qualité).
  • Spécifications fonctionnelles.

Dossier projet: plan type (2/2)

  • Spécifications techniques (dont sécurité).
  • Réalisations argumentées avec extraits de code significatifs.
  • Jeu d'essai d'une fonctionnalité représentative.
  • Veille sécurité menée pendant le projet.
  • Description d'une situation de recherche technique.

Gestion de projet attendue (abstract)

  1. Créer le Product Backlog priorisé.
  2. Estimer en story points (planning poker).
  3. Construire roadmap et jalons.
  4. Préparer sprint backlog (tâches + estimations).
  5. Suivre les écarts et replanifier.
  6. Conserver les preuves (board, Gantt, commits).

Spécifications fonctionnelles attendues

  • Cas d'usage / user stories.
  • Entrées / sorties attendues.
  • Critères d'acceptation.
  • Gestion des erreurs côté métier.
  • Matrice de traçabilité exigence → US → test.

Spécifications techniques attendues

  • Périmètre technique et stack.
  • Architecture et flux principaux.
  • Modèle de données, API et erreurs.
  • Sécurité: validation, secrets, anti-injection, traces.
  • Qualité, performance, déploiement, traçabilité.

Cycle Scrum

Refinement: méthode opérationnelle

  1. Préparer les US candidates avec contexte et critères initiaux.
  2. Clarifier avec l'équipe: questions, risques, dépendances.
  3. Découper les US trop volumineuses.
  4. Vérifier la Definition of Ready.
  5. Mettre à jour le Product Backlog avant estimation.

Exemple US (films)

### US-01 - Ajouter un film
En tant qu'administrateur
Je veux ajouter un film (titre, realisateur, annee, genre)
Afin d'enrichir le catalogue

Critere d'acceptation:
- champs valides => film enregistre
- titre vide => erreur explicite
