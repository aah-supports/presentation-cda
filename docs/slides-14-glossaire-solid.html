Glossaire SOLID

Un principe = une definition + un exemple TypeScript.

S - Single Responsibility Principle

  • Definition: une classe ne doit avoir qu'une seule raison de changer.
  • Objectif: isoler les responsabilites (metier, stockage, presentation...).
  • Impact: code plus testable, plus lisible et plus maintenable.

S - Exemple TypeScript

class PriceCalculator {
  total(items: Array<{ price: number; qty: number }>): number {
    return items.reduce((sum, i) => sum + i.price * i.qty, 0);
  }
}

class InvoicePrinter {
  print(total: number): string {
    return `Total: ${total.toFixed(2)} EUR`;
  }
}

O - Open/Closed Principle

  • Definition: ouvert a l'extension, ferme a la modification.
  • Objectif: ajouter un nouveau comportement sans re-ecrire la classe existante.
  • Lecture pratique: si tu ajoutes `VipDiscount`, tu crees une nouvelle classe, tu ne touches pas `Checkout`.

O - Exemple TypeScript

// Avant (pas OCP): il faut modifier Checkout a chaque nouveau cas
class CheckoutV1 {
  total(amount: number, customerType: "regular" | "student"): number {
    if (customerType === "student") return amount * 0.9;
    return amount;
  }
}

// Apres (OCP): on etend avec une nouvelle classe, sans modifier Checkout
interface Discount {
  apply(amount: number): number;
}

class NoDiscount implements Discount {
  apply(amount: number): number { return amount; }
}

class StudentDiscount implements Discount {
  apply(amount: number): number { return amount * 0.9; }
}

class VipDiscount implements Discount {
  apply(amount: number): number { return amount * 0.8; }
}

class Checkout {
  constructor(private discount: Discount) {}
  total(amount: number): number { return this.discount.apply(amount); }
}

const studentCheckout = new Checkout(new StudentDiscount());
const vipCheckout = new Checkout(new VipDiscount());

L - Liskov Substitution Principle

  • Definition: si le code fonctionne avec la classe parent, il doit fonctionner pareil avec n'importe quelle sous-classe.
  • Objectif: ne pas surprendre le code appelant (meme contrat, meme comportement attendu).
  • Lecture pratique: meme sequence d'appels => meme type de resultat attendu.

L - Exemple concret

class Rectangle {
  protected width = 0;
  protected height = 0;

  setWidth(w: number) { this.width = w; }
  setHeight(h: number) { this.height = h; }
  area() { return this.width * this.height; }
}

class Square extends Rectangle {
  setWidth(w: number) {
    this.width = w;
    this.height = w;
  }

  setHeight(h: number) {
    this.width = h;
    this.height = h;
  }
}

const r1: Rectangle = new Rectangle();
r1.setWidth(5);
r1.setHeight(4);
console.log(r1.area()); // 20

const r2: Rectangle = new Square(); // substitution
r2.setWidth(5);
r2.setHeight(4);
console.log(r2.area()); // 16 -> resultat inattendu pour le code appelant

L - Correction simple

interface Shape {
  area(): number;
}

class Rectangle implements Shape {
  constructor(private w: number, private h: number) {}
  area() { return this.w * this.h; }
}

class Square implements Shape {
  constructor(private size: number) {}
  area() { return this.size * this.size; }
}

// Ici, pas de substitution dangereuse via setWidth/setHeight.

I - Interface Segregation Principle

  • Definition: preferer plusieurs petites interfaces specialisees a une grosse interface.
  • Objectif: eviter d'imposer des methodes inutiles aux classes.
  • Resultat: couplage plus faible et implementations plus simples.

I - Exemple TypeScript

interface Reader {
  findAll(): Promise;
}

interface Writer {
  save(item: T): Promise;
}

class ProductQueryService {
  constructor(private reader: Reader<{ name: string }>) {}
  list() { return this.reader.findAll(); }
}

D - Dependency Inversion Principle

  • Definition: les modules metier dependent d'abstractions, pas d'implementations concretes.
  • Objectif: remplacer facilement les dependances techniques (memoire, SQL, API externe).
  • Effet: meilleure testabilite et architecture flexible.

D - Exemple TypeScript

interface ProductRepository {
  getPrice(name: string): Promise;
}

class CartService {
  constructor(private repo: ProductRepository) {}

  async total(name: string, qty: number): Promise {
    const price = await this.repo.getPrice(name);
    return price * qty;
  }
}

Recap SOLID

  • S: une responsabilite par classe.
  • O: etendre sans modifier l'existant.
  • L: sous-types substituables.
  • I: petites interfaces ciblees.
  • D: dependre d'abstractions.
